## 1. 機能概要

旅行の計画を事前に立て、スポット・予算・日程を管理する機能。
プランは作成者のみ閲覧・編集できる（非公開）。
旅行後は「この旅を記録する」フローで各スポットの投稿を作成し、プランとリンクさせることができる。
旅行完了後は完了済みフラグを立て、マイページで進行中プランと分けて管理する。

---

## 2. ユーザーストーリー

| ID | ストーリー |
|----|-----------|
| US-01 | ユーザーとして、旅行タイトル・日程・スポット・予算をまとめたプランを作成したい |
| US-02 | ユーザーとして、行きたいリストに登録したスポットをプランに簡単に追加したい |
| US-03 | ユーザーとして、スポットの訪問順序を番号で管理したい |
| US-04 | ユーザーとして、予算を項目別（交通費・宿泊費など）に入力して合計を確認したい |
| US-05 | ユーザーとして、旅行後に「この旅を記録する」から各スポットの投稿を作成したい |
| US-06 | ユーザーとして、プランから作成した投稿をプラン詳細画面でまとめて確認したい |
| US-07 | ユーザーとして、旅行が終わったらプランを完了済みにして進行中プランと分けたい |
| US-08 | ユーザーとして、誤って完了にした場合は取り消せるようにしたい |

## 3. 機能詳細

### 3.1 旅行プラン作成

- URL：`/plans/new`
- 入力項目：

| 項目 | 必須/任意 | 制約 |
|------|---------|------|
| タイトル | 必須 | 最大60文字 |
| 出発日 | 任意 | DATE 型 |
| 帰着日 | 任意 | 出発日以降であること |
| 予算内訳 | 任意 | `[{ label, amount }]` 形式。合計を `budget` に自動集計 |
| メモ | 任意 | 自由テキスト |
| スポット | 任意 | 行きたいリストから選択（複数可）。その他の投稿もアコーディオンで追加可能 |

- 処理フロー：
  1. `POST /api/plans` でプランを保存
  2. スポット一覧は `plan_spots` テーブルに `displayOrder` 付きで保存
  3. 保存成功後、作成したプランの詳細画面へ遷移

### 3.2 旅行プラン編集

- URL：`/plans/[id]/edit`
- 作成者本人のみアクセス可能（他ユーザーはマイページへリダイレクト）
- 既存データをフォームにプリセットした状態で表示する
- スポットの追加・削除・順序変更が可能

### 3.3 旅行プラン詳細

- URL：`/plans/[id]`（認証必須・作成者のみ閲覧可能）
- 表示内容：
  - タイトル・日程・メモ
  - 予算合計 + 内訳タグ（`label: amount円` 形式）
  - スポット一覧（番号付き順序表示）
  - 各スポットに「この旅を記録する →」ボタン（投稿作成画面へのリンク）
  - プランから作成された投稿の一覧（`planId` が一致する投稿）
  - 完了操作ボタン

### 3.4 「この旅を記録する」フロー

旅行プランの各スポットから投稿を作成する際の連携フロー。

- プラン詳細画面の「この旅を記録する →」をクリック（**自由入力スポット（3.6節参照）も含め、全スポットに表示される**。自由入力スポットの場合は `freeTitle`/`freeLocation`/`freeCategory` の値がプリセットされる）
- 投稿作成画面（`/posts/new`）に以下のパラメータをクエリストリングで渡す：

| パラメータ | 内容 |
|-----------|------|
| `planId` | プランID |
| `presetTitle` | スポットのタイトル |
| `presetLocation` | エリア |
| `presetCategory` | カテゴリ |
| `presetImageUrl` | スポットの画像URL（画像がある場合のみ） |

- 投稿保存後、`planId` が指定されていればプラン詳細（`/plans/[planId]`）へリダイレクトする

- 投稿作成画面でこれらをフォームにプリセット表示する
- 投稿保存時に `planId` フィールドにプランIDを記録する
- 投稿成功後、元のプラン詳細画面へリダイレクトする

### 3.5 旅行完了管理

- プラン詳細画面のチェックボックス（「未完了」/「完了済み」）をクリックすると、確認ダイアログなしで即座に `completed` の真偽値がトグルされる（2026-07-11更新：当初は確認ダイアログを挟んでいたが、いいね等の他のトグル系操作と同様に即時反映に変更）
- マイページの旅行プランタブでは、進行中プランを通常表示、完了済みプランはアコーディオン（折りたたみ）で表示
- マイページの旅行プランカウントバッジは**完了済みを除外**した件数を表示
- 完了済みプランのアコーディオン内に年度切り替えセレクト（`YearFilterBar`、「自分の投稿」「旅行レポート」タブと共通コンポーネント）を表示し、プランの`startDate`の年で絞り込める（`startDate`未設定のプランは対象外。2026-07-20追加）

### 3.6 スポットピッカー UI（2026-07-11更新）

プラン作成・編集画面のスポット選択コンポーネント。選択済みセクションの下に、以下の順で3つの候補セクションが並ぶ。

- **選択済みセクション**：選択したスポットを番号（訪問順）付きで上部に表示。✕ボタンで削除。**ドラッグ&ドロップ（`dnd-kit`）で並び替え可能**（**2件以上選択している場合のみ**ドラッグハンドルを表示。1件のみの場合は並び替えの意味がないため非表示。当初「D&Dは対象外・番号の上下ボタン等で実装」を想定していたが、実装時にドラッグ&ドロップを採用したため7節の対象外仕様から削除した）
- **🔍 スポットを検索**（候補セクションの最上部・常時展開）：キーワード入力欄が常時表示され、入力するとその場で検索結果が表示される（プロトタイプには無い、実装時に追加した機能）。**自分の投稿に限定せず、TripDiary上の全ユーザーの投稿を検索対象とする**（旅行計画は他ユーザーの投稿したスポットも対象にする方が実用的なため）。入力中は入力欄の右内側に✕ボタンが表示され、押すと検索語・検索結果をリセットする
- **🔖 行きたいリストから追加**：ユーザーの行きたいリストに登録済みの投稿を優先表示
- **その他のスポット**（アコーディオン、初期状態は折りたたみ）：開くと、検索せずに**全ユーザーの投稿を新着順に直接一覧表示**する（プロトタイプの `getPosts()` 全件表示と同じ挙動。実DBでは件数が多くなり得るため「もっと見る」でカーソルページング）
- **✏️ 新規スポット追加**（投稿が存在しないスポット向け）：TripDiary上に投稿が無いスポット（例：まだ誰も投稿していない場所）もプランに追加できるよう、スポット名（必須・60文字以内）・エリア（必須）・カテゴリ（任意）を直接入力してスポットとして追加できる。この方式で追加したスポットは投稿に紐付かない（`PlanSpot.postId` が `null` で `freeTitle`/`freeLocation`/`freeCategory` にのみ値を持つ）。プラン詳細画面では投稿参照のスポットと同様に「この旅を記録する →」が表示され、入力した内容がプリセットされた状態で投稿作成画面へ遷移する（3.4節参照）

## 4. 使用 API

| Method | エンドポイント | 説明 | 認証 |
|--------|--------------|------|------|
| GET | `/api/plans` | 自分のプラン一覧取得 | 必要 |
| POST | `/api/plans` | プラン作成 | 必要 |
| GET | `/api/plans/[id]` | プラン詳細取得 | 必要（本人のみ） |
| PUT | `/api/plans/[id]` | プラン更新 | 必要（本人のみ） |
| DELETE | `/api/plans/[id]` | プラン削除 | 必要（本人のみ） |
| PATCH | `/api/plans/[id]/complete` | 完了フラグ切り替え | 必要（本人のみ） |

**リクエスト/レスポンス例（プラン作成）**

```json
// POST /api/plans
{
  "title": "京都・奈良 2泊3日",
  "startDate": "2026-08-10",
  "endDate": "2026-08-12",
  "budgetBreakdown": [
    { "label": "交通費", "amount": 20000 },
    { "label": "宿泊費", "amount": 20000 },
    { "label": "食費", "amount": 10000 }
  ],
  "memo": "抹茶スイーツを制覇する旅",
  "spots": [
    { "type": "post", "postId": "post_abc" },
    { "type": "free", "title": "まだ投稿のない展望台", "location": "京都府", "category": "観光" }
  ]
}
```

※ `budget` はリクエストに含めず、`budgetBreakdown` の合計値を Service 層で自動算出する。`spots` は `type: "post"`（既存投稿への参照）と `type: "free"`（投稿に紐付かない自由入力スポット）の混在配列。

## 5. バリデーション仕様

| 項目 | フロントエンド | バックエンド |
|------|-------------|------------|
| タイトル | 必須・最大60文字 | 必須・最大60文字 |
| 帰着日 | 出発日以降であること | 出発日以降であること |
| 予算内訳 | 金額 > 0 またはラベルが入力済みの項目のみ保存 | 配列形式・各要素は `label(string)` + `amount(int)` |
| スポット（`type: "post"`） | 任意・存在する投稿IDのみ | 存在確認・自分の行きたいリスト外も可 |
| スポット（`type: "free"`） | スポット名必須・最大60文字、エリア必須、カテゴリ任意 | 同左（Zodスキーマで検証） |
| 完了操作 | - | 本人のみ実行可能 |

---

## 6. 制約・前提条件

- プランは作成者のみ閲覧・編集・削除できる（他ユーザーには非公開）
- `budgetBreakdown` の空項目（金額0かつラベル空）は保存時に除外する
- スポット（`type: "post"`）は投稿データへの参照であり、投稿が削除された場合はプランから自動除外する（CASCADE）。`type: "free"` のスポットは投稿を参照しないため対象外
- 同一プランに同一投稿を複数回スポットとして登録することは意図的に許容している（重複防止は行わない）
- プランを削除しても紐付く投稿の `planId` は NULL になるだけで投稿は削除されない

---

## 7. 対象外仕様

| 機能 | 理由 |
|------|------|
| プランの公開・シェア | 非公開設計のためスコープ外 |
| 複数ユーザーでのプラン共同編集 | スコープ外 |
| プランのテンプレート保存 | スコープ外 |
