# 認証機能定義書

**バージョン:** 1.2
**作成日:** 2026-06-27
**更新日:** 2026-06-28

---

## 1. 機能概要

Auth.js v5（next-auth 5.0.0-beta.31）を使用したトークンベースの認証。
メールアドレス + パスワードによる Credentials Provider で実装する。
セッションは JWT Strategy を採用し、署名済み JWT を HttpOnly Cookie に保存する（DB へのセッション書き込みなし）。

---

## 2. ユーザーストーリー

| ID | ストーリー |
|----|-----------|
| US-01 | 新規ユーザーとして、ニックネーム・メール・パスワードを入力して登録したい |
| US-02 | 登録済みユーザーとして、メール・パスワードでログインしたい |
| US-03 | ログイン中のユーザーとして、ログアウトしてセッションを終了したい |
| US-04 | 未認証ユーザーとして、認証が必要なページにアクセスしたらログイン画面にリダイレクトされたい |

---

## 3. 機能詳細

### 3.1 ユーザー登録

- 入力項目：ニックネーム・メールアドレス・パスワード・パスワード確認
- 処理フロー：
  1. フロントエンドでバリデーション
  2. `POST /api/auth/signup` でメール重複チェック
  3. パスワードを bcrypt でハッシュ化して保存
  4. 登録成功後、自動的にログイン状態にする
- 異常系：
  - メール重複 → 409 エラー「このメールアドレスはすでに使用されています」
  - バリデーションエラー → 400 エラー

### 3.2 ログイン

- 入力項目：メールアドレス・パスワード
- UI：パスワード入力欄の右端にトグルボタン（👁）を配置。押している間だけ平文表示（`type="text"`）し、離すと非表示（`type="password"`）に戻る
- 処理フロー：
  1. Auth.js の `signIn('credentials', ...)` を呼び出す
  2. メールアドレスでユーザー検索
  3. bcrypt でパスワード照合
  4. 照合成功時に JWT を生成し HttpOnly Cookie に保存
- 異常系：
  - メール未存在 or パスワード不一致 → 「メールアドレスまたはパスワードが正しくありません」

### 3.3 ログアウト

- Auth.js の `signOut()` を呼び出す
- HttpOnly Cookie（JWT）を削除する
- ログイン画面にリダイレクトする

### 3.4 認証ガード（ミドルウェア）

- `proxy.ts` で認証が必要なルートを保護する（Next.js 16 より `middleware.ts` から改名）
- 未認証ユーザーが保護ルートにアクセスした場合、`/login` にリダイレクトする
- 認証済みユーザーが `/login` / `/signup` にアクセスした場合、`/` にリダイレクトする

**保護するルート（未認証でアクセスすると `/login` にリダイレクト）**

| ルート | 理由 |
|-------|------|
| `/mypage` | マイページ（認証ユーザー専用） |
| `/posts/new` | 投稿作成（ログイン必須） |
| `/posts/[id]/edit` | 投稿編集（本人確認が必要） |
| `/settings` | プロフィール編集（本人確認が必要） |
| `/plans/new` | 旅行プラン作成（ログイン必須） |
| `/plans/[id]` | 旅行プラン詳細（作成者のみ閲覧可能） |
| `/plans/[id]/edit` | 旅行プラン編集（作成者のみ） |

**公開ルート（未認証でもアクセス可能）**

| ルート | 備考 |
|-------|------|
| `/` | 探索ページ（全体公開フィード） |
| `/posts/[id]` | 投稿詳細（閲覧のみ） |
| `/users/[id]` | ユーザープロフィール（閲覧のみ） |
| `/tags/[tag]` | エリアタグ絞り込み |
| `/search` | ユーザー検索 |
| `/login` / `/signup` | 認証画面 |

## 4. 使用 API

| Method | エンドポイント | 説明 | 認証 |
|--------|--------------|------|------|
| POST | `/api/auth/signup` | ユーザー登録 | 不要 |
| GET / POST | `/api/auth/[...nextauth]` | Auth.js セッション管理（ログイン・ログアウト・セッション取得） | - |

---

## 5. バリデーション仕様

| 項目 | フロントエンド | バックエンド |
|------|-------------|------------|
| ニックネーム | 必須・1〜20文字 | 必須・1〜20文字 |
| メールアドレス | 必須・メール形式 | 必須・メール形式・重複チェック |
| パスワード | 必須・8文字以上 | 必須・8文字以上 |
| パスワード確認 | 必須・パスワードと一致 | - |

---

## 6. セキュリティ仕様

| 項目 | 仕様 |
|------|------|
| パスワード | bcrypt（ソルトラウンド：12）でハッシュ化して保存 |
| セッション | JWT Strategy（HttpOnly Cookie に署名済み JWT を保存） |
| セッション有効期限 | 30日（`maxAge: 30 * 24 * 60 * 60`）。アクティブ利用中は毎日自動延長（`updateAge: 24 * 60 * 60`） |
| AUTH_SECRET | 環境変数で管理。本番は EC2 の `.env.local` に設定 |

---

## 7. 制約・前提条件

- Auth.js v5 beta を使用するため、GA リリース後に破壊的変更が生じる可能性がある
- OAuth（Google・GitHub 等）によるソーシャルログインは対象外
- パスワードリセット機能は対象外

---

## 8. 対象外仕様

| 機能 | 理由 |
|------|------|
| OAuth ソーシャルログイン | スコープ外（将来的に追加可能） |
| パスワードリセット | スコープ外 |
| メール認証（確認メール送信） | スコープ外 |
| 2段階認証 | スコープ外 |

## 9. E2E テスト

`e2e/auth.spec.ts` に以下のテストケースを実装済み。

| # | テストケース | 確認内容 |
|---|------------|---------|
| 1 | 新規登録 → ダッシュボード遷移 | 登録後に `/dashboard` に遷移し、ニックネームが表示される |
| 2 | ログイン → ダッシュボード遷移 | 正しい認証情報でログインすると `/dashboard` に遷移する |
| 3 | 誤パスワード → エラーメッセージ表示 | 「メールアドレスまたはパスワードが正しくありません」が表示される |
| 4 | 未認証で `/dashboard` → `/login` にリダイレクト | 未認証アクセスが `/login` にリダイレクトされる |

**実行方法：**
```bash
pnpm exec playwright test e2e/auth.spec.ts --headed
```
