## 0. 位置づけ

Phase 2.5-C（開発基盤整備）で導入した構造化ログの方針と、ログを使った運用監視・障害対応の考え方をまとめる。

同コース内の別課題 TimeLine（Spring Boot）で出された要件「構造化ログの作成（バックエンドのみ対象）＋ログの運用監視・障害対応を踏まえた設計書作成（監視項目・閾値はアプリ内容に即して設計、アラート設計自体は不要）」を、TripDiary（Next.js Route Handler）向けに翻訳したものである。

**対象範囲**: バックエンド（Route Handler / Service / Repository）のログのみ。フロントエンドのログ（ブラウザコンソール等）は対象外。
**対象外**: 実際のアラート発報設定（閾値監視ツールの具体的な設定作業）。本書はあくまで設計であり、Phase 3以降で機能が増えるたびに随時更新する前提。

## 1. 構造化ログ方針

### 1.1 使用ライブラリ

`pino`（開発時は `pino-pretty` で整形表示）。`src/lib/logger.ts` にsingletonとして実装。

### 1.2 ログレベル定義

| レベル | 用途 |
|---|---|
| `error` | 未捕捉の例外（500系）。スタックトレースを含む |
| `warn` | 想定内のエラー（4xx系: 404 NotFound / 403 Forbidden / 400 Validation / 409 Conflict）。クライアント起因のため深刻度は低いが、頻発時は異常の兆候になり得る |
| `info` | （現状未使用。将来的にリクエスト単位のアクセスログ等に利用を検討） |
| `debug` | 開発環境のデフォルトレベル。詳細なデバッグ情報 |

環境変数 `LOG_LEVEL` で上書き可能。未設定時は本番 `info` ／開発 `debug`。

### 1.3 出力項目

`src/lib/api-error.ts` の `handleApiError` を唯一の差し込み口とし、以下を構造化フィールドとして出力する。

| フィールド | 内容 |
|---|---|
| `errorType` | エラークラス名（`UnauthorizedError` / `NotFoundError` / `ForbiddenError` / `ValidationError` / `ConflictError` / 未分類は概要のみ） |
| `message` | エラーメッセージ（4xx系） |
| `details` | バリデーションエラーのフィールド単位詳細（`ValidationError` のみ） |
| `err` | 未捕捉例外（500系）のエラーオブジェクト一式（スタックトレース含む、pinoの `err` シリアライザを利用） |

※ 現状 `endpoint`（どのAPIパスで発生したか）・`userId`・`requestId` は自動付与していない。`handleApiError` は呼び出し元のRoute Handlerの情報を持たないため、将来的にミドルウェアや各Route Handler側での明示的な付与を検討する（2.5-Cのスコープでは見送り）。

### 1.4 出力先

現状は標準出力（開発: `pino-pretty` で整形、本番: PM2ログとして収集）。

`docs/インフラ構成書.md` では「アプリログ: PM2ログ／未定（CloudWatch等は実装時に検討）」とされている。本設計はその「未定」を埋めるものではなく、まず構造化ログの**中身**（何を出すか）を先に固めた段階。出力先をCloudWatch等の集約基盤に送る配線は、実際にPM2運用を始めるデプロイフェーズ（実装計画書 Phase 6）で改めて検討する。

## 2. 監視項目と閾値（TripDiaryのアプリ内容に即した設計）

閾値は暫定値。実際のトラフィック傾向を計測してから調整する前提（★実測値なしの仮設定）。

| 監視項目 | 説明 | 暫定閾値 | 関連エンドポイント |
|---|---|---|---|
| APIエラー率（5xx） | 未捕捉例外の発生率。バグや依存サービス障害の兆候 | 直近5分間で5%超 | 全API |
| 認証失敗率 | ログイン試行の失敗率。ブルートフォース攻撃の兆候にもなり得る | 同一IPから短時間（1分）に5回超の失敗 | `auth/[...nextauth]`, `auth/signup` |
| バリデーションエラー率（400） | 特定エンドポイントで急増した場合、フロントとバックエンドのスキーマ不整合を疑う | 直近5分間で同一エンドポイントへの400が20%超 | 全API（特に `posts`, `posts/[id]/comments`, `plans`） |
| 画像アップロード失敗率 | ディスク容量枯渇やファイルシステム書き込み不可の間接指標 | 直近5分間で10%超 | `upload/post` |
| DB接続エラー | Prisma接続エラー（`PrismaClientKnownRequestError` の接続系コード等） | 発生したら即座に検知対象（0件が正常） | 全API（DB経由の処理） |
| レスポンスタイム悪化 | p95レイテンシの悪化。DB遅延やN+1の再発を検知 | p95が平常時の3倍を超える | 特に一覧・集計系（`posts`, `posts/portal`, `posts/explore`, `plans`, `stats`）。`stats` はService層でJS側集計（1クエリ+メモリ内グルーピング）を行うため、投稿数の多いユーザーでの悪化を優先的に監視する |
| 他人リソースへの不正アクセス試行率（403） | 他ユーザーのプラン等への操作試行。総当たり的なID推測の兆候 | 直近5分間で同一IPからの403が10回超 | `plans/[id]`, `plans/[id]/complete` |

## 3. 障害対応フロー（インシデントレスポンス）

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  1. 検知    │ --> │ 2. 一次切り分け│ --> │  3. 対応    │ --> │ 4. 復旧確認 │ --> │ 5. 事後記録 │
└─────────────┘     └──────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
```

### 1. 検知
- ログ（`error`/`warn`レベル）から異常値を把握。現状は手動でログを確認する運用（自動アラートは本書のスコープ外）
- 2節の監視項目のいずれかが閾値を超えた場合を「異常」とみなす

### 2. 一次切り分け
- `errorType` で分類: `ValidationError`/`ConflictError`（クライアント起因、緊急度低） vs 未分類の500系（サーバー起因、緊急度高）
- 該当エンドポイント・発生時刻を特定し、直前のデプロイ・DBマイグレーションとの相関を確認

### 3. 対応（既知エラー分類ごとの初動）
| 分類 | 初動 |
|---|---|
| ValidationError急増 | フロントの入力チェックとバックエンドのZodスキーマの不整合を疑い、該当フォームを確認 |
| DB接続エラー | RDS/MySQLコンテナの稼働状況、コネクションプール枯渇を確認 |
| 500（未分類） | ログの `err.stack` からスタックトレースを特定し、該当コードを確認。再現手順があれば `docs/テスト設計書.md` に沿ってテストケースを追加 |
| 認証失敗率急増 | ブルートフォース攻撃の可能性。レート制限の状況を確認（実装計画書 Phase 5-C 参照） |

### 4. 復旧確認
- 該当エンドポイントへの正常系リクエストが成功することを確認
- ログのエラー発生率が閾値内に戻ったことを確認

### 5. 事後記録
- 原因・対応内容・再発防止策を記録する（現状は決まった記録先がないため、当面はPR説明やコミットメッセージに残す。将来的に障害対応ログを一元管理する場所を検討）

## 4. エラー監視（Sentry等SaaS連携）の設計（Phase 2.5-D）

Phase 2.5-Dでは、Sentry等のエラー監視SaaSの**実導入は行わず、連携ポイントの実装と送信設計のみ**を行う。ログ（1〜3節）が「サーバーに残る記録を後から見に行く」ものであるのに対し、エラー監視SaaSは「本番でエラーが起きた瞬間に自動で検知・通知する」ものであり、両者は役割が異なる（4.4参照）。

### 4.1 実装済みの連携ポイント

| 層 | 場所 | 状態 |
|---|---|---|
| バックエンド | `src/lib/monitoring.ts` の `captureException()` を `src/lib/api-error.ts` の未捕捉例外（500系）パスから呼び出す | 実装済み（DSN未設定時は no-op） |
| フロントエンド | `src/app/global-error.tsx`（画面クラッシュ捕捉）にコメントで送信口を明示 | 差し込み口のみ用意 |

`captureException()` は唯一の差し込み口であり、監視SaaS導入時はこの関数の中身を差し替えるだけで全体に反映される。

### 4.2 送信データの設計（送るもの／送らないもの）

| 送る | 送らない |
|---|---|
| エラーオブジェクト（スタックトレース） | リクエストボディの中身（投稿本文・コメント本文等の入力内容） |
| 発生箇所（`source`、例: `"api/posts POST"`） | パスワード・セッショントークン等の認証情報 |
| ログインユーザーの**IDのみ**（`userId`） | メールアドレス・ニックネーム等の個人を特定しうる情報 |

個人情報保護の観点から、ユーザーを一意に追跡するのに必要な最小限（ID）に留める。本文やメール等は送らない。

### 4.3 環境分離

- 送信の有無は環境変数 `SENTRY_DSN`（サーバー）／`NEXT_PUBLIC_SENTRY_DSN`（クライアント）の有無で切り替える
- **DSN未設定（開発環境・現状の本番）では送信しない**（`captureException` が即 return する no-op）。これにより開発中に余計な送信が発生せず、本番でDSNを設定した時点でのみ監視が有効になる
- DSNは `.env` に置き、`.env.sample` にキーだけ（値なし）を追記する（機密情報のコミット防止）

### 4.4 監視項目との役割分担（2節との対応）

エラー監視SaaS導入後、2節の監視項目のうち以下は「ログの手動確認」から「SaaSによる自動検知」へ移行できる。

| 2節の監視項目 | ログ（手動） | SaaS導入後（自動） |
|---|---|---|
| APIエラー率（5xx） | `error` ログを目視 | `captureException` 経由で自動通知・件数集計 |
| DB接続エラー | `error` ログを目視 | 同上（未捕捉例外として捕捉される） |
| 認証失敗率 / バリデーション400率 | `warn` ログを目視 | **SaaSには送らない**（4xxは送信対象外）ため引き続きログで確認 |
| レスポンスタイム悪化 | — | SaaSのパフォーマンス監視機能（APM）で計測（導入時に別途検討） |

つまり、SaaSは「サーバー起因の異常（5xx）の自動検知」を担い、クライアント起因（4xx）やレイテンシは引き続きログ／別手段で見る、という分担になる。

### 4.5 導入手順（将来、ユーザー確認後に実施）

1. Sentry等でアカウント作成 → Next.jsプロジェクトを登録し **DSN** を取得
2. `@sentry/nextjs` 等のSDKを導入し、`.env` に `SENTRY_DSN` / `NEXT_PUBLIC_SENTRY_DSN` を設定
3. `src/lib/monitoring.ts` の `captureException` に実際の送信処理を実装（4.2の送信データ設計に従う）
4. `src/app/global-error.tsx` にクライアント側の送信を実装
5. 意図的にエラーを発生させ、SaaSのダッシュボードに届くことを確認

※ アカウント作成・DSN発行は外部サービス契約を伴うため、自動では行わずユーザーの実施・確認を前提とする。
