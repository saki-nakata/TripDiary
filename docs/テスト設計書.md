## 1. テスト方針・目的

Phase 3以降でAPIエンドポイントが急増する前に、単体・統合テストの基盤を整備する（Phase 2.5-B）。
本番影響を避けつつ、権限チェック・バリデーション境界値・DB連携ロジックといった不具合が起きやすい箇所を優先的にカバーする。

同コース内の別課題 TimeLine（Spring Boot）のテスト実装（Controller/Service/Repositoryの3層戦略、`対象_条件_期待結果` の命名規則）を参考にしつつ、TripDiary（Next.js + Prisma + MySQL）向けに翻訳して適用した。

## 3. 使用ツール・環境

- **テストランナー**: Vitest 3.2.6（`@vitejs/plugin-react` は vitest 3系との互換性の都合上 4.7.0 に固定）
- **フロントエンド**: `@testing-library/react` / `@testing-library/jest-dom` / `@testing-library/user-event`、環境は `jsdom`（ファイル先頭に `// @vitest-environment jsdom` を指定）
- **バックエンド**: 環境は `node`（デフォルト）
- **E2E（既存）**: `@playwright/test`
- **ORM/DB**: Prisma 5.22 + MySQL 8.0（テスト用コンテナ）
- **カバレッジ**: `@vitest/coverage-v8`
- **環境変数読み込み**: `dotenv`（`vitest.config.ts` 起動時に `.env.test` を読み込み、`dotenv-cli` で Prisma CLI 実行時にも同様に注入）

---

## 4. テストデータ・DB戦略

- `docker-compose.yml` に `mysql-test` サービスを追加（ポート3307、DB名 `tripdiary_test`、`tmpfs` マウントで永続化しない使い捨てDB）
- `.env.test`（gitignore対象）で `DATABASE_URL` をテスト用DBに向ける。開発用DB（`.env` → ポート3306）とは完全に分離
- スキーマ適用: `pnpm prisma:migrate:test`（`dotenv-cli` で `.env.test` を注入して `prisma migrate deploy`）
- Repository層テストは各 `describe` の `beforeEach` でテーブルをクリーンアップ（`SET FOREIGN_KEY_CHECKS=0` で外部キー制約を一時解除し関連テーブルを `deleteMany`）してからデータを投入する方式を採用。Prisma+MySQLではトランザクション分離のみで安全に隔離するのが難しいため、テストごとの明示的クリーンアップとした
- 本番用DBには一切接続しないため、誤って本番データに影響するリスクを構造的に排除

## 5. テスト観点

- **正常系**: 期待するステータスコード・レスポンス形状
- **異常系**: 401（未認証）・403（他人のリソース操作）・404（存在しないID）・400（バリデーション）・409（メール重複）
- **境界値分析**: 例）`title` 40文字（正常）/ 41文字（異常）、`rating` 1・5（正常）/ 0・6（異常）、`body` 2000文字（正常）/ 2001文字（異常）、`nickname` 20文字（正常）/ 21文字（異常）、`password` 8文字（正常）/ 7文字（異常）
- **同値分割**: カテゴリの妥当な値／不正な値、confirmPasswordの一致／不一致
- **権限・認可**: 本人/他人の分岐を両方検証し、権限NG時にrepositoryの更新系関数が**呼ばれないこと**まで確認する

---

## 6. 層別戦略と命名規則

| 層 | 手法 | モック対象 |
|---|---|---|
| Controller（Route Handler） | ブラックボックス・薄い結合テスト | `@/lib/auth`（`vi.mock`）、依存するservice/repository |
| Service | ホワイトボックス・純粋単体 | repository（`vi.mock`） |
| Repository | 実DB検証 | モックしない（テスト用MySQLに対して実行） |

**命名規則**: `対象_条件_期待結果`（例: `createPost_タイトル40文字_201`、`updatePost_他人の投稿_403`）。境界値は正常・異常のペアで書く。対象メソッドごとに `// ─── 対象名 ───` のコメント区切りでグルーピングする。

## 10. CI実行フロー（将来像）

Phase 5-BでCI（GitHub Actions等）を導入する際、以下をPRトリガーで自動実行する想定:

1. `pnpm install`
2. `pnpm lint`
3. `pnpm typecheck`（`tsc --noEmit`）
4. テスト用MySQLコンテナを起動 → `pnpm prisma:migrate:test` → `pnpm test`
5. （任意）`pnpm test:coverage` の結果をアーティファクトとして保存
6. 既存 `e2e/auth.spec.ts`（Playwright）は別ジョブとして分離実行

現時点（Phase 2.5）ではCI設定ファイルは作成せず、ローカル実行の確認のみとする。
