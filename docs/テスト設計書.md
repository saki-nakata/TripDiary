## 1. テスト方針・目的

Phase 3以降でAPIエンドポイントが急増する前に、単体・統合テストの基盤を整備する（Phase 2.5-B）。
本番影響を避けつつ、権限チェック・バリデーション境界値・DB連携ロジックといった不具合が起きやすい箇所を優先的にカバーする。

同コース内の別課題 TimeLine（Spring Boot）のテスト実装（Controller/Service/Repositoryの3層戦略、`対象_条件_期待結果` の命名規則）を参考にしつつ、TripDiary（Next.js + Prisma + MySQL）向けに翻訳して適用した。

## 3. 使用ツール・環境

- **テストランナー**: Vitest 4.1.10（`@vitejs/plugin-react` 6.0.3 + `vite` 8.1.4。導入時はvite 8がpostcss未リリースバージョンを要求し一時的に3.2.6/4.7.0へ固定していたが、postcssリリース後に最新版へ更新済み）
- **フロントエンド**: `@testing-library/react` / `@testing-library/jest-dom` / `@testing-library/user-event`、環境は `jsdom`（ファイル先頭に `// @vitest-environment jsdom` を指定）
- **バックエンド**: 環境は `node`（デフォルト）
- **E2E**: `@playwright/test`（`reporter: [html, json, list]` を設定、`playwright-report/` にレポート出力）
- **ORM/DB**: Prisma 6.19 + MySQL 8.0（テスト用コンテナ）
- **カバレッジ**: `@vitest/coverage-v8`
- **環境変数読み込み**: `dotenv`（`vitest.config.ts` 起動時に `.env.test` を読み込み、`dotenv-cli` で Prisma CLI 実行時にも同様に注入）

## 4. テストデータ・DB戦略

- `docker-compose.yml` に `mysql-test` サービスを追加（ポート3307、DB名 `tripdiary_test`、`tmpfs` マウントで永続化しない使い捨てDB）
- `.env.test`（gitignore対象）で `DATABASE_URL` をテスト用DBに向ける。開発用DB（`.env` → ポート3306）とは完全に分離
- スキーマ適用: `pnpm prisma:migrate:test`（`dotenv-cli` で `.env.test` を注入して `prisma migrate deploy`）
- Repository層テストは各 `describe` の `beforeEach` でテーブルをクリーンアップ（`SET FOREIGN_KEY_CHECKS=0` で外部キー制約を一時解除し関連テーブルを `deleteMany`）してからデータを投入する方式を採用。Prisma+MySQLではトランザクション分離のみで安全に隔離するのが難しいため、テストごとの明示的クリーンアップとした
- 本番用DBには一切接続しないため、誤って本番データに影響するリスクを構造的に排除
- **⚠️ 2026-07-06 追記**: Repository層テストファイルが複数存在する場合、Vitestの既定（ファイル間並列実行）のままだと各ファイルの `beforeEach` クリーンアップが共有テーブル（`users`等）に対して競合し、他ファイルが投入中のデータを消してしまい外部キー制約違反で失敗することがある（Phase 3-C で `follow.repository.test.ts` を追加した際に発覚）。`vitest.config.ts` に `fileParallelism: false` を設定し、テストファイルを直列実行することで回避した

## 5. テスト観点

- **正常系**: 期待するステータスコード・レスポンス形状
- **異常系**: 401（未認証）・403（他人のリソース操作）・404（存在しないID）・400（バリデーション）・409（メール重複）
- **境界値分析**: 例）`title` 40文字（正常）/ 41文字（異常）、`rating` 1・5（正常）/ 0・6（異常）、`body` 2000文字（正常）/ 2001文字（異常）、`nickname` 20文字（正常）/ 21文字（異常）、`password` 8文字（正常）/ 7文字（異常）
- **同値分割**: カテゴリの妥当な値／不正な値、confirmPasswordの一致／不一致
- **権限・認可**: 本人/他人の分岐を両方検証し、権限NG時にrepositoryの更新系関数が**呼ばれないこと**まで確認する

---

## 6. 層別戦略と命名規則

| 層 | 手法 | モック対象 |
|---|---|---|
| Controller（Route Handler） | ブラックボックス・薄い結合テスト | `@/lib/auth`（`vi.mock`）、依存するservice/repository |
| Service | ホワイトボックス・純粋単体 | repository（`vi.mock`） |
| Repository | 実DB検証 | モックしない（テスト用MySQLに対して実行） |

**命名規則**: `対象_条件_期待結果`（例: `createPost_タイトル40文字_201`、`updatePost_他人の投稿_403`）。境界値は正常・異常のペアで書く。対象メソッドごとに `// ─── 対象名 ───` のコメント区切りでグルーピングする。

## 10. E2E安定化の知見（2026-07-06、Phase 3で判明）

Phase 3で `users.spec.ts` を追加した際、既存の `posts.spec.ts` を含む全体実行が断続的に失敗する事象を詳細調査した。原因は当初「タイムアウトが短い」ことを疑ったが、実際には以下の**2つの異なる根本原因**であり、タイムアウト延長では解決しないことが判明した。

### 10.1 モジュールレベル変数のワーカー間非共有

`posts.spec.ts`・`users.spec.ts` は同一 `describe` 内のテスト間で `let postId = ""` のようなモジュールレベル変数を共有し、「同じファイルの全テストは同じワーカープロセスで実行される」という前提に依存していた。

通常はこの前提が成り立つが、**あるテストが失敗すると、Playwrightが後続テストを別のワーカープロセスに再割り当てすることがある**。新しいワーカーはモジュールを最初から読み込むため、共有変数は初期値（空文字）に戻る。この状態で `page.goto('/posts/' + postId)` を呼ぶと `/posts/`（実質 `/posts`）にアクセスし、404が発生する。

`playwright-report/results.json` の各テスト結果に含まれる `workerIndex` を比較することで、削除テストだけ別ワーカーで実行されていたことを直接確認できた（調査時の実例: テスト1〜4が`workerIndex: 1`、テスト5（削除）のみ`workerIndex: 3`）。

**対処**: `test.describe(...)` を `test.describe.serial(...)` に変更。これは「同一ワーカーでの順次実行」と「1つ失敗したら後続はスキップする」ことを明示的に保証するPlaywrightの公式パターンで、ワーカー再割り当てによる状態喪失が構造的に発生しなくなる。

### 10.2 `CommentSection.tsx` のレースコンディション（Phase 1由来の既存バグ）

上記の対処後も「コメント投稿→一覧に反映される」が稀に失敗するケースが残っていた。調査の結果、`CommentSection.tsx` の初回コメント一覧取得（`useEffect` 内の `loadComments()`）が、投稿（POST）の完了より**後に解決すると**、無条件の `setComments(data.comments)` が投稿前の空リストで上書きしてしまうレースコンディションと判明した。人間の操作速度では顕在化しないが、Playwrightの高速な自動操作では初回GETの完了を待たずに投稿が完了することがあり、負荷時にはさらに発生しやすくなる。

**対処**: `opSeqRef`（操作の連番）を導入し、`loadComments`・`handleSubmit`・`handleDelete` の開始時にインクリメント。`loadComments`のfetch完了時に連番が変わっていれば（＝自分より新しい操作が開始していれば）、コメント一覧・カーソル等のデータ反映のみを破棄する（ローディング解除は常に行う）。回帰テストを `CommentSection.test.tsx` に追加した（T-56）。

この過程で、投稿・コメント双方に同じ「削除」ボタンが存在する場合にPlaywrightのロール検索が曖昧になる問題も判明したため、`DeleteButton.tsx` に `data-testid="delete-post-button"` を追加してテストを安定させた。

**教訓**: ①タイムアウトを延ばす対処は、真の原因（状態管理・並行処理のバグ）を覆い隠すことがある。②E2Eで再現する不具合は、実装側の実際のバグを暴くことがあるため、テスト側だけでなく実装側も疑って調査する。
